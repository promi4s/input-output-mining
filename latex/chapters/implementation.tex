\section{Discovering the Role Functions Automatically}
\label{chap:finding-initial-final}



In our method, we assumed that for an input OCEL the functions $role_{in}: E \times O \rightarrow \{in, init, \bot\}$ and $role_{out}: E \times O \rightarrow \{out, final, \bot\}$ are given.
We provide a method to discover them automatically.
This section presents the implementation of this functionality.



This is achieved by exploiting the total order of events based on timestamps.
In the simplest case, we assume that an object is an output of an event if it is related to a later event, and an input if it is related to a previous event.
If there are two events $e, e' \in E$ with $time(e) < time(e')$ and an object $o \in O$ related to both, meaning $(e, o) \in E2O$ and $(e', o) \in E2O$, we set an output on the previous event, $role_{out}(e, o) = out$, and an input on the later event, $role_{in}(e', o) = in$.
The total order ensures that this is valid.
These definitions also immediately satisfy the consistency condition of the role functions.



To discover the initial inputs and final outputs in an event, we use a case distinction based on the nature of the event.
Events are categorized as logistical or transformational.


\paragraph{Logistical events.}
An event is a logistical event if it semantically reflects movement, storage, or change of state of associated objects, all of which persist until after the event.

\paragraph{Transformational events.}
An event is a transformational event if it semantically represents a consumption of the inputs to create a new output, such that the inputs might not exist in their original form after the event.

Objects that still have an unknown role from the fallback case are rendered with a double-arrow dashed arc in the Petri net, or the user can provide their role as a hint to the algorithm.



\label{chap:impl-using-mfa}
Alternatively, the remaining objects with an unknown role can be resolved using MFA if they have a mass attribute.
Accordingly, we assume that the mass-balance principle holds.
Then, the objects with an unknown role are assigned initial input or final output roles in each possible combination, and the assignment with the smallest difference between the total input and output mass is chosen.
This corresponds to the closest match of the total input and output mass.
This procedure is equivalent to minimizing the process-wide waste $procwaste$ from \cref{chap:annotating-the-petri-net} over the possible assignments.




\section{Ocelescope Plugin}



In this section, we present how a user can use the method and implementation in Ocelescope~\autocite{ocelescope}.
It is implemented in the \emph{Object Flow Miner} plugin.
This plugin offers four methods that run the object flow miner, which differ in using weighted or variable arcs and having waste annotations or not.
This menu is shown in \autoref{fig:impl-method-menu}.
Each one gives an \emph{object flow net}.



\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth, height=\textheight, keepaspectratio]{figures/impl-method-menu.png}
    \caption{The available methods in our Ocelescope plugin.}
    \label{fig:impl-method-menu}
\end{figure}


As an example, choosing \enquote{Weighted arcs} leads to the configuration screen in \autoref{fig:impl-nonannotated}.
It offers a range of parameters:
\begin{enumerate}[label={\circled{\arabic*}}]
    \item The OCEL2.0 event log.
    
    \item 
    A list of object types that are filtered out prior to running the object flow miner.

    \item 
    The role function maps these object types to always be an initial input.
    This allows the user to manually set the role functions.
    For object types that are not given here or in the next field, the heuristic algorithm is applied to determine the role function.

    \item 
    The role function maps these object types to always be a final output.

    \item 
    If an attribute $mass$ exists, selecting this checkbox causes an MFA to be performed as described in ~\cref{chap:impl-using-mfa} to approximate mass balance as closely as possible.

    \item 
    The threshold $\alpha$, which filters configurations of the same activity, as described in \cref{chap:filtering}.

    \item 
    The threshold $\beta$, which filters the succeeding activities in a place and therefore reduces the number of arcs, also described in \cref{chap:filtering}.

    \item 
    Selecting this checkbox also displays the inputs and outputs of each place and transition as additional information in each node, which is done throughout the examples in our method.
\end{enumerate}
The Ocelescope application automatically populates each field with appropriate suggestions, such as exactly the object types that occur in the log.
Clicking on \texttt{Submit} executes the algorithm, yielding the same OFN model that was also the result of our method in \cref{chap:creating-the-model}, see \autoref{fig:algorithm-petrinet-step}.
Ocelescope shows a preview of the model, see \autoref{fig:impl-unannotated-result}, which can be made full-screen and downloaded.

We also explain the parameters for the \enquote{Weighted arcs with annotations} method, shown in \autoref{fig:impl-annotated}.
In addition to the previously described parameters, now the interface includes the options:
\begin{enumerate}[start=9, label={\circled{\arabic*}}]
    \item
    These object types are counted in each place where they are initial input or final output.

    \item
    The mass of these object types is calculated for each place where they are initial input or final output.

    \item
    The attribute $mass$ is averaged and summed for each configuration of the activities listed here.

    \item
    The difference between the number of input and output objects is calculated for each configuration of the activities listed here.

    \item
    The difference between the summed masses of input and output objects is calculated for each configuration of the activities listed here.
\end{enumerate}
With the given configuration, this generates the model with waste annotations shown as an example in our method in~\autoref{fig:algorithm-waste-annotated}.
